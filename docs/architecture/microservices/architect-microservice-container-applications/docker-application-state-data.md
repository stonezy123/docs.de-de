# <a name="state-and-data-in-docker-applications"></a><span data-ttu-id="ce4ca-104">Zustand und Daten in Docker-Anwendungen</span><span class="sxs-lookup"><span data-stu-id="ce4ca-104">State and data in Docker applications</span></span>

<span data-ttu-id="ce4ca-105">In den meisten Fällen können Sie sich einen Container als Instanz eines Prozesses vorstellen.</span><span class="sxs-lookup"><span data-stu-id="ce4ca-105">In most cases, you can think of a container as an instance of a process.</span></span> <span data-ttu-id="ce4ca-106">Ein Prozess behält keinen persistenten Zustand bei.</span><span class="sxs-lookup"><span data-stu-id="ce4ca-106">A process doesn't maintain persistent state.</span></span> <span data-ttu-id="ce4ca-107">Während ein Container in den lokalen Speicher schreiben kann, gleicht die Annahme, dass eine Instanz auf unbestimmte Zeit vorhanden ist, der Annahme, dass ein Speicherort im Arbeitsspeicher dauerhaft ist.</span><span class="sxs-lookup"><span data-stu-id="ce4ca-107">While a container can write to its local storage, assuming that an instance will be around indefinitely would be like assuming that a single location in memory will be durable.</span></span> <span data-ttu-id="ce4ca-108">Sie müssen davon ausgehen, dass Containerimages ebenso wie Prozesse über mehrere Instanzen verfügen oder letztendlich gelöscht werden.</span><span class="sxs-lookup"><span data-stu-id="ce4ca-108">You should assume that container images, like processes, have multiple instances or will eventually be killed.</span></span> <span data-ttu-id="ce4ca-109">Wenn sie mit einem Containerorchestrator verwaltet werden, müssen Sie davon ausgehen, dass sie zwischen Knoten oder virtuellen Computern verschoben werden.</span><span class="sxs-lookup"><span data-stu-id="ce4ca-109">If they're managed with a container orchestrator, you should assume that they might get moved from one node or VM to another.</span></span>

<span data-ttu-id="ce4ca-110">Die folgenden Lösungen werden verwendet, um Daten in Docker-Anwendungen zu verwalten:</span><span class="sxs-lookup"><span data-stu-id="ce4ca-110">The following solutions are used to manage data in Docker applications:</span></span>

<span data-ttu-id="ce4ca-111">Über den Docker-Host als [Docker-Volumes](https://docs.docker.com/engine/admin/volumes/):</span><span class="sxs-lookup"><span data-stu-id="ce4ca-111">From the Docker host, as [Docker Volumes](https://docs.docker.com/engine/admin/volumes/):</span></span>

- <span data-ttu-id="ce4ca-112">**Volumes** werden in einem Bereich des Hostdateisystems gespeichert, der von Docker verwaltet wird.</span><span class="sxs-lookup"><span data-stu-id="ce4ca-112">**Volumes** are stored in an area of the host filesystem that's managed by Docker.</span></span>

- <span data-ttu-id="ce4ca-113">**Einbindungen über „Bind“** können jedem Ordner im Hostdateisystem zugeordnet werden, sodass der Zugriff nicht vom Docker-Prozess gesteuert werden kann – dies wäre ein Sicherheitsrisiko, weil ein Container auf vertrauliche Betriebssystemordner zugreifen könnte.</span><span class="sxs-lookup"><span data-stu-id="ce4ca-113">**Bind mounts** can map to any folder in the host filesystem, so access can't be controlled from Docker process and can pose a security risk as a container could access sensitive OS folders.</span></span>

- <span data-ttu-id="ce4ca-114">**Einbindungen über tmpfs** sind ähnlich wie virtuelle Ordner, die nur im Arbeitsspeicher des Hosts vorhanden sind und nie in das Dateisystem geschrieben werden.</span><span class="sxs-lookup"><span data-stu-id="ce4ca-114">**tmpfs mounts** are like virtual folders that only exist in the host's memory and are never written to the filesystem.</span></span>

<span data-ttu-id="ce4ca-115">Über den Remotespeicher:</span><span class="sxs-lookup"><span data-stu-id="ce4ca-115">From remote storage:</span></span>

- <span data-ttu-id="ce4ca-116">[Azure Storage](https://azure.microsoft.com/documentation/services/storage/), das geografisch verteilbaren Speicher und eine geeignete Lösung für langfristige Persistenz für Container bereitstellt.</span><span class="sxs-lookup"><span data-stu-id="ce4ca-116">[Azure Storage](https://azure.microsoft.com/documentation/services/storage/), which provides geo-distributable storage, providing a good long-term persistence solution for containers.</span></span>

- <span data-ttu-id="ce4ca-117">Relationale Remotedatenbanken wie die [Azure SQL-Datenbank](https://azure.microsoft.com/services/sql-database/) oder NoSQL-Datenbanken wie [Azure Cosmos DB](/azure/cosmos-db/introduction) oder Zwischenspeicherdienste wie [Redis](https://redis.io/).</span><span class="sxs-lookup"><span data-stu-id="ce4ca-117">Remote relational databases like [Azure SQL Database](https://azure.microsoft.com/services/sql-database/) or NoSQL databases like [Azure Cosmos DB](/azure/cosmos-db/introduction), or cache services like [Redis](https://redis.io/).</span></span>

<span data-ttu-id="ce4ca-118">Über den Docker-Container:</span><span class="sxs-lookup"><span data-stu-id="ce4ca-118">From the Docker container:</span></span>

- <span data-ttu-id="ce4ca-119">**Overlay File System**.</span><span class="sxs-lookup"><span data-stu-id="ce4ca-119">**Overlay File System**.</span></span> <span data-ttu-id="ce4ca-120">Dieses Docker-Feature implementiert ein Copy-on-Write-Task, der aktualisierte Informationen im Stammdateisystem des Containers speichert.</span><span class="sxs-lookup"><span data-stu-id="ce4ca-120">This Docker feature implements a copy-on-write task that stores updated information to the root file system of the container.</span></span> <span data-ttu-id="ce4ca-121">Diese Information befindet sich „über“ dem ursprünglichen Image, auf dem der Container basiert.</span><span class="sxs-lookup"><span data-stu-id="ce4ca-121">That information is "on top" of the original image on which the container is based.</span></span> <span data-ttu-id="ce4ca-122">Wenn der Container aus dem System gelöscht wird, gehen diese Änderungen verloren.</span><span class="sxs-lookup"><span data-stu-id="ce4ca-122">If the container is deleted from the system, those changes are lost.</span></span> <span data-ttu-id="ce4ca-123">Aus diesem Grund ist es zwar möglich, den Zustand eines Containers in seinem lokalen Speicher zu speichern, doch das Entwerfen eines umgehenden Systems würde mit der Prämisse des Containerdesigns kollidieren, das standardmäßig zustandslos ist.</span><span class="sxs-lookup"><span data-stu-id="ce4ca-123">Therefore, while it's possible to save the state of a container within its local storage, designing a system around this would conflict with the premise of container design, which by default is stateless.</span></span>

<span data-ttu-id="ce4ca-124">Allerdings stellt die Verwendung von Docker-Volumes mittlerweile das bevorzugte Verfahren zum Verwalten von lokalen Daten in Docker dar.</span><span class="sxs-lookup"><span data-stu-id="ce4ca-124">However, using Docker Volumes is now the preferred way to handle local data in Docker.</span></span> <span data-ttu-id="ce4ca-125">Wenn Sie weitere Informationen zu Speichern in Container benötigen, lesen Sie [Docker storage drivers](https://docs.docker.com/storage/storagedriver/select-storage-driver/) (Docker-Speichertreiber) und [About storage drivers](https://docs.docker.com/storage/storagedriver/) (Über Speichertreiber).</span><span class="sxs-lookup"><span data-stu-id="ce4ca-125">If you need more information about storage in containers check on [Docker storage drivers](https://docs.docker.com/storage/storagedriver/select-storage-driver/) and [About storage drivers](https://docs.docker.com/storage/storagedriver/).</span></span>

<span data-ttu-id="ce4ca-126">Nachfolgend erhalten Sie weitere Details zu diesen Optionen:</span><span class="sxs-lookup"><span data-stu-id="ce4ca-126">The following provides more detail about these options:</span></span>

<span data-ttu-id="ce4ca-127">**Volumes** sind Verzeichnisse, die vom Hostbetriebssystem den Verzeichnissen in Containern zugeordnet werden.</span><span class="sxs-lookup"><span data-stu-id="ce4ca-127">**Volumes** are directories mapped from the host OS to directories in containers.</span></span> <span data-ttu-id="ce4ca-128">Wenn Code im Container Zugriff auf das Verzeichnis hat, ist dies ein Zugriff auf ein Verzeichnis auf dem Hostbetriebssystem.</span><span class="sxs-lookup"><span data-stu-id="ce4ca-128">When code in the container has access to the directory, that access is actually to a directory on the host OS.</span></span> <span data-ttu-id="ce4ca-129">Dieses Verzeichnis ist nicht an die Lebensdauer des Containers selbst gebunden, und es wird von Docker verwaltet und von der Kernfunktionalität des Hostcomputers isoliert.</span><span class="sxs-lookup"><span data-stu-id="ce4ca-129">This directory is not tied to the lifetime of the container itself, and the directory is managed by Docker and isolated from the core functionality of the host machine.</span></span> <span data-ttu-id="ce4ca-130">Deshalb sind diese Datenvolumes dazu entworfen, Daten unabhängig von der Lebensdauer des Containers beizubehalten.</span><span class="sxs-lookup"><span data-stu-id="ce4ca-130">Thus, data volumes are designed to persist data independently of the life of the container.</span></span> <span data-ttu-id="ce4ca-131">Wenn Sie einen Container oder ein Image aus dem Docker-Host löschen, werden die persistenten Daten im Datenvolume nicht gelöscht.</span><span class="sxs-lookup"><span data-stu-id="ce4ca-131">If you delete a container or an image from the Docker host, the data persisted in the data volume isn't deleted.</span></span>

<span data-ttu-id="ce4ca-132">Volumes können benannt oder anonym sein (letzteres ist die Standardeinstellung).</span><span class="sxs-lookup"><span data-stu-id="ce4ca-132">Volumes can be named or anonymous (the default).</span></span> <span data-ttu-id="ce4ca-133">Benannte Volumes sind eine Weiterentwicklung von **Datenvolumecontainern** und vereinfachen die gemeinsame Nutzung von Daten durch Container.</span><span class="sxs-lookup"><span data-stu-id="ce4ca-133">Named volumes are the evolution of **Data Volume Containers** and make it easy to share data between containers.</span></span> <span data-ttu-id="ce4ca-134">Volumes unterstützen auch Volumetreiber, mit denen Sie u.a. Daten auf Remotehosts speichern können.</span><span class="sxs-lookup"><span data-stu-id="ce4ca-134">Volumes also support volume drivers that allow you to store data on remote hosts, among other options.</span></span>

<span data-ttu-id="ce4ca-135">**Einbindungen über „Bind“** sind schon lange verfügbar und ermöglichen die Zuordnung jedes Ordners zu einem Bereitstellungspunkt in einem Container.</span><span class="sxs-lookup"><span data-stu-id="ce4ca-135">**Bind mounts** are available since a long time ago and allow the mapping of any folder to a mount point in a container.</span></span> <span data-ttu-id="ce4ca-136">Einbindungen über „Bind“ weisen mehr Einschränkungen als Volumes sowie einige erhebliche Sicherheitsprobleme auf, daher sind Volumes die empfohlene Option.</span><span class="sxs-lookup"><span data-stu-id="ce4ca-136">Bind mounts have more limitations than volumes and some important security issues, so volumes are the recommended option.</span></span>

<span data-ttu-id="ce4ca-137">**Einbindungen über tmpfs** sind im Grunde virtuelle Ordner, die nur im Arbeitsspeicher des Hosts vorhanden sind und nie in das Dateisystem geschrieben werden.</span><span class="sxs-lookup"><span data-stu-id="ce4ca-137">**tmpfs mounts** are basically virtual folders that live only in the host's memory and are never written to the filesystem.</span></span> <span data-ttu-id="ce4ca-138">Sie sind schnell und sicher, verbrauchen aber Arbeitsspeicher und sind nur für temporäre, nicht persistente Daten vorgesehen.</span><span class="sxs-lookup"><span data-stu-id="ce4ca-138">They are fast and secure but use memory and are only meant for temporary, non-persistent data.</span></span>

<span data-ttu-id="ce4ca-139">Wie in Abbildung 4-5 gezeigt, können reguläre Docker-Volumes außerhalb der Container selbst aber innerhalb der physischen Grenzen des Hostservers oder der VM gespeichert werden.</span><span class="sxs-lookup"><span data-stu-id="ce4ca-139">As shown in Figure 4-5, regular Docker volumes can be stored outside of the containers themselves but within the physical boundaries of the host server or VM.</span></span> <span data-ttu-id="ce4ca-140">Docker-Container können jedoch nicht von einem Hostserver oder virtuellen Computer aus auf ein Volume auf einem anderen Hostserver oder virtuellen Computer zugreifen.</span><span class="sxs-lookup"><span data-stu-id="ce4ca-140">However, Docker containers can't access a volume from one host server or VM to another.</span></span> <span data-ttu-id="ce4ca-141">Anders gesagt: Bei diesen Volumes ist es nicht möglich, Daten zu verwalten, die von auf verschiedenen Docker-Hosts ausgeführten Containern gemeinsam verwendet werden. Dieses Szenario ließe sich allerdings mit einem Volumetreiber umsetzen, der Remotehosts unterstützt.</span><span class="sxs-lookup"><span data-stu-id="ce4ca-141">In other words, with these volumes, it isn't possible to manage data shared between containers that run on different Docker hosts, although it could be achieved with a volume driver that supports remote hosts.</span></span>

<span data-ttu-id="ce4ca-151">**Azure Storage.**</span><span class="sxs-lookup"><span data-stu-id="ce4ca-151">**Azure Storage.**</span></span> <span data-ttu-id="ce4ca-152">Geschäftsdaten müssen in der Regel in externen Ressourcen oder Datenbanken wie Azure Storage abgelegt werden.</span><span class="sxs-lookup"><span data-stu-id="ce4ca-152">Business data usually will need to be placed in external resources or databases, like Azure Storage.</span></span> <span data-ttu-id="ce4ca-153">Azure Storage bietet konkret die folgenden Dienste in der Cloud:</span><span class="sxs-lookup"><span data-stu-id="ce4ca-153">Azure Storage, in concrete, provides the following services in the cloud:</span></span>
